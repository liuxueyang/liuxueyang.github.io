1 使用 Hexo 生成靜態博客，使用多說評論系統

幾年前的事情了，大概從 2014 年開始？

  1. 各種主題選得頭疼，前端又不會，只能靠魔改別人的模板、改改配置才能維持生活這樣子。
  2. 維護、同步有點兒麻煩。其實也不是特別麻煩啦，也就是兩個部分——一部分源碼，另外一部分是生成的靜態博客。
  3. 有點兒討厭 Markdown 語法，比如我在 Emacs 裏面寫文章的時候，經常會用 M-q 來讓一個段落保持在 80 個字符寬度內，看起來整齊一些。然後生成的 HTML 裏就會在換行處多出一個空格。當然 org 導出的 HTML 文件也有這個問題啦，不過可以很愉快地解決就是了。

2 切換成 VimWiki

大概只用了一段時間吧。其實我是不太會用啦。大概是因爲不是特別喜歡在 Vim 裏面輸入中文，畢竟三種模式已經夠了，還要加上切換中英文。。

3 切換成 Jekyll

看起來是更簡潔一些吧。 Github Pages 原生支持。然而還是需要維護各種配置哦，還是要用討厭的 Markdown 。個人感覺好像還不如用 Hexo 舒服。

4 切換成 Hugo

把原來的別的平臺上的博客都轉成 Markdown 格式，打算都放在這裏，一共大概有四百篇博客。發現用 Hexo 生成 HTML 就要好久「大概幾分鐘？」。這個有點兒不能忍受啊，然後就考慮切換成 Hugo 。可以說優勢很明顯了，四百篇左右的 Markdown ，生成 HTML 大概是秒出。

  1. 還是需要選亂七八糟、各式各樣的博客模板，修改配置，同步配置等等。
  2. 找不到容易使用又滿足自己需求的模板，前端還是不會。。只保持勉強能夠使用的狀態。

5 使用 org-mode 直接生成 HTML

沒有主題可選，沒有配置可選，只能夠依靠找別人的網站，「借鑑」（偷）點兒東西過來這樣子。

雖然頁面沒有之前好看了，但是不用維護那些討厭的配置了。相關的配置裏面就在 .emacs 裏面了。 org 源文件在一個私有 repo 裏面維護，生成的 HTML 在另外一個獨立的 repo 裏面維護。除了一個 CSS 文件、還有 Favicon ，別的都是 org 文件。
